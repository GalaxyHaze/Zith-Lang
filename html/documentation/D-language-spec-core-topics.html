Zith Spec Core Topics

This document maps the language specification (docs/Zith-spec.md) into documentation-friendly sections.

1) Introduction & Philosophy

Zith is a statically typed, compiled, general-purpose language with a deliberately small, stable core and a large, specialised toolbox. See Overview.

  • Explicit over implicit semantics.
  • Safety by default through Node Resource Analysis (NRA) — no borrow checker, no GC.
  • Universal applicability through contexts and data-oriented patterns.
  • The Rule of Three: if a function or block uses more than three specialised tools, rethink the design.

2) Node Resource Analysis (NRA)

NRA models values as nodes and bindings as edges in a graph analysed at compile time. See Memory Model — NRA.

  • Tracks connectivity, validity, and move state.
  • Distinguishes origins: literal, allocator, stack, view.
  • Performs branch-aware liveness checks for failable/optional outcomes.
  • Four rules: argument exclusivity, no dead node access, no escaping belong, lend behavioural promise.

3) Ownership Keywords (Five Modes)

Zith defines six ownership/borrowing modes:

  • default — owned, exclusive (no keyword needed).
  • lend — exclusive mutable temporary borrow.
  • view — read-only non-owning reference.
  • unique — exclusive ownership and transfer.
  • share — compile-time-proven shared mutable state.
  • belong — parent-child structural relationship (back-pointers).

4) Type System

The specification combines nominal and structural typing. See Type System.

  • Primitives, slices/arrays, user types, generics, and trait-oriented capabilities.
  • Optional/failable navigation: ?T, T!, ?, !, or.
  • Pattern matching with when and union narrowing with is.

5) Module System & Visibility

See Module System.

  • import, from, export, alias, use.
  • Visibility: pub, mod, mod(N), mod(..).
  • Type constraints with or vs union separators with ,.

6) Functions & Closures

See Functions.

  • Six function kinds: fn, const fn, async fn, async<Generator> fn, flow fn, raw fn.
  • Closures as packs: |capture| (params) { body }.
  • Dynamic levels: static vs dyn, opaque packs.
  • @ prefix for macros, bare names for functions.

7) Control Flow

See Control Flow.

  • Parentheses required on conditions; English logical keywords (and, or, not).
  • for, if/else, when pattern matching, -> chain flow.
  • flow fn with markers, docks, and jumps.

8) Error Handling

Fully static, return-based. See Error Handling.

  • ?T (optional) and T! (result) with ? / ! propagation.
  • or fallback, with/catch, fail blocks, throw.
  • must (debug panic) vs raw (always unchecked).

9) Data-Oriented Architecture

See ECS & Scenes.

  • Components as POD data, scenes as memory regions (arenas).
  • Scene transition replaces memory; custom allocators supported.
