WASM Build Requirements

To compile zithc to WebAssembly so the Playground can run Zith code in the browser, these items need to be addressed:

Toolchain

# Requirement Details
1 Emscripten SDK emsdk toolchain for compiling C++ to WASM. Use emcmake and emmake wrappers.
2 POSIX guards Wrap unistd.h, termios.h, isatty with __EMSCRIPTEN__ guards (already done for _WIN32, extend pattern).
3 File system shim CompilationSession reads source files from disk — needs MEMFS or a virtual FS that accepts source strings from JS instead.
4 C API entry point The existing zithc-capi.cpp is the right boundary. Expose create_from_buffer() and run() via EMSCRIPTEN_BINDINGS.
5 No threads Ensure solver/sema don't assume std::thread. WASM is single-threaded by default.
6 No exceptions to JS Compile with -s DISABLE_EXCEPTION_CATCHING=1. The CAPI already uses nothrow new — extend to all C++/WASM boundary code.
7 Memory budget Set -s TOTAL_MEMORY=64MB initially with ALLOW_MEMORY_GROWTH=1.
8 Export functions Use EMSCRIPTEN_KEEPALIVE for zithc_compile(code) -> {output, errors}. Return JSON strings for easy JS parsing.
9 CMake toolchain Add cmake/emscripten.cmake toolchain file, selectable via -DCMAKE_TOOLCHAIN_FILE=....
10 CI build Add WASM target to build-artifact.yml, publish .wasm + .js glue as artifacts.

Frontend Integration

  • Editor — a <textarea> or contenteditable div with monospace font and basic syntax highlighting via JS.
  • Run button — calls the exported WASM zithc_compile function with the editor contents.
  • Output panel — a read-only area showing stdout, stderr, and compile errors returned by the WASM module.
  • Module loading — load the .wasm file with WebAssembly.instantiateStreaming and the generated JS glue.
