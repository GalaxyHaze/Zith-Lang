Release v0.2.0 — The LSP Awakening

21 June 2026

Summary

This is the one. The Zith Language Server finally comes alive, bringing real IDE superpowers to VS Code. Hover over a symbol and see its type. Press Ctrl+Space and get completions — including import path suggestions. Hit Ctrl+Shift+O and navigate your file through a structured symbol tree, with children. Make a typo? The LSP will whisper "did you mean?" right in your editor. And if you hover over something that isn't quite valid yet — don't worry, the hover still shows you what it thinks it is.

We also shipped a brand new VS Code extension that bundles it all together: the LSP auto-downloads on first activation, the standard library is fetched automatically (no credentials needed), and you get syntax highlighting, snippets, task runners, and problem matchers out of the box. Install once, start coding.

Highlights

  • Hover That Works — hover over any symbol and see its declaration info. Functions show their full signature (fn greet(name: str): str). Structs, enums, unions, traits, type aliases, and variables all show their kind and name. And if you hover over a typo, the LSP won't just give up — it falls back to basic info and even suggests what you might have meant via Levenshtein distance.
  • Doc Comments on Hover — /// and /** */ doc comments are now captured during scanning and displayed when you hover over any declaration. Finally, your documentation has a home.
  • Did-You-Mean Quick Fix — undefined identifier diagnostics now embed Levenshtein suggestions in the data.suggestions field. The lightbulb pops up with one-click replacement CodeActions that replace the misspelled token with the suggested name.
  • Smart Completion — completions work in two modes: in normal code you get local + imported symbols (fully type-checked when possible); inside import/from/export statements you get live filesystem-driven module path suggestions (std/, std/io/console, etc).
  • Document Symbols with Children — the outline view finally shows nested symbols. Struct fields, enum variants, function parameters, union variants, and methods all appear as proper children. No more flat lists.
  • Auto-Download Everything — the extension downloads the LSP binary from GitHub releases (or falls back to a local build) and fetches the standard library from the public GalaxyHaze/Zith repo, all on first activation. No manual setup.
  • Resilient Hover — even on incomplete or broken code, the LSP doesn't bail out. It runs to TypeChecked when it can (for full type info), but falls back gracefully to the symbol table when parsing fails mid-edit. Your editor stays useful no matter what state your code is in.

Technical Details

Added

  • Doc comment storage — SymbolData::doc_span field captures the span of /// or /** */ tokens preceding each declaration during the scan phase. Doc comments are no longer discarded.
  • Code Actions infrastructure — textDocument/codeAction handler and provider in the LSP, wired into the dispatch loop and exposed as "codeActionProvider": {"codeActionKinds": ["quickfix"]} in capabilities
  • Did-you-mean quick fix — provideCodeActions reads embedded suggestions from diagnostic data.suggestions and emits one quickfix per Levenshtein match, with a TextEdit replacing the misspelled token.
  • Doc comment storage — SymbolData::doc_span field captures the span of /// or /** */ tokens preceding each declaration during the scan phase. Doc comments are no longer discarded.
  • Doc comment hover rendering — hover.cpp extracts doc comment text from the source span, strips /// and /** */ markers, and renders as Markdown above the declaration signature.
  • C API suggestions — zithc_diag_suggestion_count() and zithc_diag_suggestion_get() expose the compiler's Levenshtein suggestion engine to C consumers, enabling the LSP to embed suggestions in diagnostic JSON without re-running the compiler
  • Doc comment capture in parser scan phase — consecutive Docs tokens are merged into a single span and passed to syms.declare() for top-level fn, struct, enum, union, component, trait, and interface declarations
  • Import path completion — scanStdlibPath() in features/completion.cpp walks the stdlib filesystem for module path suggestions, prefix-filtered and deduplicated
  • Auto-download stdlib — download-lsp.js fetches stdlib/ from the public GalaxyHaze/Zith repo Contents API to server/stdlib/; extension calls ensureStdlib() on activation
  • LSP binary auto-download — 3-tier fallback: GitHub release → local copy → source build, with clear failure message as last resort
  • Stdlib path forwarding — stdlibPath_ read from initializationOptions.stdlibPath in onInitialize, passed to diagnostics, hover, and completion providers
  • C API — zithc_session_add_include_dir() exposes include directory support for the LSP's use

Changed

  • Hover resilience — runTo(TypeChecked) failure no longer aborts hover; basic symbol info (kind + name) always shows, type-expr details render conditionally only when type-checking succeeded
  • JSON-RPC respond() — removed is_null() guard so every response includes a "result" field, fixing malformed responses (hover on unknown tokens previously sent empty envelopes)
  • DocumentSymbol children — replaced hardcoded "children": [] with recursive JSON serialization; children resolved via linear sym-id scan and decl_id match map
  • spanToRange — removed the span.end <= span.start guard that zeroed all ranges (root cause: mixed byte-offset and token-index units in ScanEntry.span). Now computes single-point range from start position only.
  • symKindToLspSymbolKind — fixed out-of-range LSP symbol kind values for Alias, Variable, Module, Trait, and Enum
  • Import error positions — compiler compilation-session.cpp:271 uses import->span instead of {}, so import errors point to the import/from/export keyword instead of line 1
  • Parser scan phase — 7 syms.declare() calls patched with name spans for child symbols (fn params, struct fields, enum variants, union variants, struct/trait methods)
  • Stdlib path config removed — zith.lsp.stdlibPath VS Code setting eliminated; path is auto-detected from server/stdlib/

Fixed

  • Hover returning malformed JSON-RPC — respond() no longer omits the result field for null results
  • DocumentSymbol positions stuck at (1,1) — spanToRange no longer zeroes all spans; now computes correct single-point ranges from token start
  • DocumentSymbol children always empty — children are now built from symbol table entries via decl_id matching and SymbolData.members iteration
  • Hover fails on incomplete code — runTo(TypeChecked) failure no longer aborts the entire hover; symbol-table fallback always provides basic info
  • Stdlib download 403 — User-Agent header added to GitHub Contents API requests

Release v0.1.3

20 June 2026

Summary

This release introduces the ZIR bytecode IR subsystem, completes the migration from std::vector to DynArray in two core subsystems, adds Span fields on all AST nodes, and ports the ZIR interpreter from the prototype branch. The compiler now has proper arena-backed containers throughout ZIR and Diagnostics, source-location tracking on every AST node, and a complete stack-based bytecode interpreter ready for pipeline integration.

Highlights

  • ZIR Bytecode IR — 26 opcodes, bytecode emitter (HIR → ZIR), stack-based interpreter, block/function/module types. Ported from prototype and adapted for DynArray.
  • AST Span coverage — every ExprNode, StmtNode, and DeclNode variant now carries a memory::Span for source location tracking
  • std::vector → DynArray — ZIR structs (blocks, functions, modules) and Diagnostics (labels, suggestions) now use arena-backed DynArray
  • ZirInterpreter register file — fixed 16-slot std::array replaces std::vector for predictable performance
  • Pratt parser foundation — span tracking in parser prepares for complete expression/statement parsing in v0.2.0.0
  • Version roadmap — comprehensive plan from v0.2.0.0 through v0.8.0.0 with all 4 codegen paths documented

Release v0.3.2

20 June 2026

Summary

This release focuses on infrastructure and code quality — we've extracted core utilities into a standalone zith-util repo, replaced all std::unordered_map usage with a custom open-addressing FlatMap, and added Span fields to every AST node in preparation for LSP support. The compiler is now free of STL hash maps.

Highlights

  • zith-util extracted — Optional, Result, Arena, DynArray, and FlatMap moved to their own GitHub repo for independent reuse
  • FlatMap replaces std::unordered_map — open-addressing hash map with Fibonacci hashing, transparent string lookup, installed across 4 subsystems
  • AST Span coverage — every ExprNode, StmtNode, and DeclNode variant now carries a memory::Span for source location tracking
  • Zero STL hash maps — <unordered_map> and <unordered_set> completely eliminated from the compiler source
  • LSP Preparation — span fields lay the groundwork for go-to-definition, hover, and diagnostics with precise source locations

Technical Details

Added

  • zith-util repo (GalaxyHaze/zith-util) — extracted headers for optional.hpp, result.hpp, arena.hpp, dyn_array.hpp, flat_map.hpp with their own CMake INTERFACE library target and README
  • src/memory/flat_map.hpp — open-addressing FlatMap<Key, Value> with Fibonacci hashing, transparent std::string/std::string_view lookup, iterator support, and STL-like API
  • Span field to all AST nodes — LitValue, BinaryNode, UnaryNode, CallNode, IdentNode, FieldNode, IndexNode, RangeNode, BlockNode, IfNode, WhileNode, LetNode, AssignNode, RetNode, FnDeclNode, StructDeclNode, EnumDeclNode, UnionDeclNode, ComponentDeclNode, TraitDeclNode, InterfaceDeclNode, ImportNode

Changed

  • StringInterner — std::unordered_map<std::string_view, InternedId> replaced with FlatMap
  • SourceMap — file path cache switched from std::unordered_map<std::string, FileId> to FlatMap
  • ImportManager — index_by_path_ and resolving_ (formerly std::unordered_set) both migrated to FlatMap
  • TypeLower — generic context pointer changed from std::unordered_map<std::string_view, TypeId>* to FlatMap*
  • AstBuilder — all builder methods now accept an optional memory::Span parameter
  • Parser — expression, statement, and declaration builders now pass computed token spans to the AST nodes

Fixed

  • vector<bool> specialization — FlatMap with bool value triggers std::vector<bool> bit-vector proxy reference issue; resolving_ changed from FlatMap<string, bool> to FlatMap<string, char>
  • Missing unordered_set include — import-manager.cpp still used std::unordered_set<size_t> for merge visited set; include restored locally

Release v0.3.1

19 June 2026

Summary

This release focuses on stability and infrastructure — we've introduced a Type Walker utility for safely traversing the type graph, added a Platform Abstraction layer for cleaner cross-platform code, and fixed several critical bugs including undefined behavior in the token stream, a dangling span in TypeFn, a TOCTOU race in SourceMap, and null-pointer hazards in Optional. The test suite has also grown significantly with 6 new test files.

Highlights

  • Type Walker — new walkSubTypes() utility for recursively visiting sub-types, replacing manual switch-cases in unification and future passes
  • Platform Abstraction — platform.hpp centralizes OS-specific defines, reducing #ifdef clutter across the codebase
  • Critical Bug Fixes — 7 security/correctness fixes in a single sweep (see Technical Details)
  • Unifier Refactor — occurs() check now uses walkSubTypes instead of hand-written type-kind dispatch
  • 6 New Test Suites — diagnostic engine advanced, error codes lookup, platform, result, string interner, and resolver tests

Release v0.3.0

13 June 2026

Summary

This release introduces the Name Resolution System — a dedicated pipeline stage between import resolution and semantic analysis. The new Resolver module walks the AST after all imports are known, resolves every identifier to its SymId, and records results in a side table that the Sema pipeline consumes directly. We've also extended the parser with range syntax (..), arrow member access (->), and fat arrows (=>), added a RangeNode to the AST, fixed several diagnostic edge cases, and made the compiler portable to Windows.

Highlights

  • Name Resolution Pipeline Stage — new Resolve stage between import and sema, with qualified and unqualified name resolution
  • Alias Resolution — import aliases (import foo as f) are resolved through the symbol table with target tracking
  • Improved Diagnostics — specific error codes for non-namespace access (NotNamespace) and missing members (NoMember)
  • Parser Extensions — range operator .., arrow access ->, fat arrow =>
  • AST RangeNode — new expression node with lhs/rhs sub-expressions, wired through builder and printer
  • Windows Support — _WIN32 guards added to 5 CLI command files for cross-platform compatibility
  • Comprehensive Resolver Tests — 10 test cases covering unqualified, qualified, scope, alias, and edge-case resolution

Technical Details

Added

  • Full Resolver implementation (src/import/resolver.*) — qualified and unqualified name lookup, scope management, alias following, side table recording
  • Stage::Resolved enum value in pipeline plan — new pipeline stage between Imported and TypeChecked
  • resolveStage() in CompilationSession — instantiates Resolver, runs it, extracts resolved table for sema
  • RangeNode AST node with lhs/rhs expression fields, builder method, and printer support
  • SymOrigin tracking in ImportManager — records which file and local symbol each merged symbol originates from (originOf() query)
  • SymbolTable::declareInScope() — declare a symbol in a specific scope (not just current)
  • Error codes NotNamespace (2005) and NoMember (2006) with human-readable descriptions
  • Windows _WIN32 guards in tool.cpp, run.cpp, project.cpp, interactive.cpp, deps.cpp
  • Resolver::takeResolvedTable() — moves out the resolved symbol side table for consumption by later stages
  • SemaPipeline::resolved_ pointer — optional reference to the resolver side table, checked before fallback lookup in visitIdent
  • 10 resolver unit tests in tests/unit/resolver-test.cpp

Changed

  • Resolved table integration — visitIdent checks the resolver side table first, falls back to syms().lookup(), and skips duplicate diagnostics when the resolver already reported an error
  • Import alias declaration now stores the target SymId in the symbol's target field, enabling alias chain resolution
  • Symbol table SymbolData struct — added SymId target field and updated constructor signature (8 parameters)
  • declare / declareInScope accept optional SymId target parameter
  • Parser — tryCompoundOp extended for =>, .. range detection before . field access, -> arrow member access
  • Diagnostics emit — fixed Optional(nullptr) to {}, added bounds checks on d.primary.end, clamped unsigned subtraction
  • visitIf in SemaPipeline — creates proper HirBranch node instead of discarding the then_branch value
  • CompilationSession constructor — initializes resolved_syms_ from sym_arena_
  • Import merge re-exports — restored duplicate-checking declare_re_export lambda

Fixed

  • Diagnostic emit — null optional crash in emit.cpp when optional holds nullptr
  • Diagnostic emit — bounds check on d.primary.end before access
  • Import merge — re-export path was silently overwriting duplicates instead of reporting DuplicateDecl
