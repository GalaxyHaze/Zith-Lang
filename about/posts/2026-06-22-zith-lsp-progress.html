zith-lsp Progress — Hover, Position, Diagnostics

2026-06-22

The zith-lsp server has been the focus this week, and it's finally taking shape. Most of the groundwork lives in the CAPI layer of zithc itself — the LSP binary is a thin wrapper that talks to zithcLib over a C ABI.

What's Working Now

Hover. zithc_hover() takes a session and a byte offset, finds the identifier at that position, and returns a markdown-formatted function signature. This means an editor can show type information on mouseover without re-parsing the source.

Position conversion. The zithc_offset_to_position() function converts byte offsets to 0-based line/column pairs — essential for LSP protocol compatibility.

Diagnostics with intelligence. The heuristic engine has been integrated into the LSP path. When you have an undefined identifier, the compiler not only reports the error but suggests the closest match via Levenshtein distance. For ambiguous calls or no-matching-function errors, it lists all candidate signatures so you can see what went wrong.

Cross-file support. The merge_sym() function now preserves declaration IDs, spans, kinds, targets, and documentation spans across import boundaries. This is the data foundation for go-to-definition and find-references — the LSP just needs to be wired through.

Build System

The LSP builds alongside the compiler in CI. The macOS build produces a universal binary (x86_64 + ARM64), and the Windows build works with both MSVC and clang-cl. The CI caches have been cleaned up to avoid the corruption issues we were seeing before.

What's Next

Completions and go-to-definition are the next targets. The symbol table and cross-file merging infrastructure is ready — it's a matter of wiring the LSP protocol messages.
